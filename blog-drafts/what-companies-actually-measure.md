# What Companies Actually Measure About AI-Assisted Development (And What They Don't)

Every major tech company has published metrics about their AI coding tools. The pattern across all of them is striking — and it reveals the delivery gap more clearly than any single study.

## The Table

| Company | What They Publish | What They Don't |
|---------|------------------|-----------------|
| **Spotify** | 1,500+ merged PRs from Honk, 60-90% time savings | No change failure rate. No rework rate. No escaped defects. |
| **Stripe** | CI wait time reduced 70%, 20% more features shipped | Almost nothing about AI coding impact specifically. |
| **Uber** | uReview covers 90% of 65K weekly diffs, 75% "useful" | No data on whether AI review catches defects that escape. "Useful" is self-reported. |
| **Dropbox** | AI user cohorts have higher PR throughput | No CFR, no rework, no cost-per-change. Acknowledges they haven't connected to business outcomes yet. |
| **Anthropic** | 200K transcripts analyzed, PR output ~200% YoY | No CFR, no rework, no escaped defects. Most rigorous methodology, still input-focused. |
| **Google** | "25% of all new code is AI-generated" (Pichai, earnings call) | Zero methodology. No quality comparison. Investor relations, not engineering measurement. |
| **Microsoft** | 55% faster task completion, 46% code generated by Copilot | No CFR comparison. No rework. GitClear found 4x code clone growth — Microsoft hasn't addressed it. |
| **Amazon** | "4,500 developer-years saved" | No methodology for the claim. No CFR. No quality comparison. |
| **Shopify** | CEO ran autoresearch loop, cultural mandate | Zero systematic engineering metrics. Pure CEO showmanship. |
| **Booking.com** | 16% higher PR merge rate for AI users, DX Core 4 framework | Names CFR as a dimension but hasn't published the numbers. |
| **Coinbase** | 40% AI-generated code, tracks bugs/incidents internally | Won't disclose whether bugs went up or down since AI adoption. |
| **OpenAI** | 70% more PRs merged weekly, "quality maintained" | "Quality maintained" has no supporting data. |

## The Pattern

Every company measures generation. Almost nobody measures whether it worked.

**What nearly everyone tracks:**
- PR throughput / merge rate
- Percentage of code that is AI-generated
- Developer time saved (self-reported or estimated)
- Adoption rate (daily/weekly active users)
- Task completion speed

**What almost nobody tracks (or publishes):**
- Rework rate — how often code gets revised after merge
- Change failure rate by origin — does AI code cause more production incidents?
- Escaped defects — bugs that reach users, segmented by AI vs human
- Cost per accepted change — total cost including review burden, rework, incidents
- Abandoned work — code that gets started but thrown away
- Review burden — how much additional review time AI-generated code demands

## What The Independent Data Shows

The independent data tells a different story from the vendor metrics.

**Cortex 2026 Benchmark:** PRs per author up 20% YoY, but incidents per PR up 23.5% and change failure rates up ~30%. Their report title: "AI Is Making Engineering Faster, But Not Better."

**Faros AI (10,000+ developers):** 98% more PRs merged, but 154% larger PRs, 91% longer review times, and net throughput improvement of zero. The correlation between AI adoption and performance "evaporates at the company level."

**Jellyfish (2M+ PRs):** 60% of engineering leaders cite "lack of clear metrics" as their biggest AI challenge. Only 20% of teams use engineering metrics to measure AI impact at all.

**METR (randomized controlled trial):** Experienced open-source developers were 19% slower with AI tools — despite believing they were 20% faster.

## Why This Matters

The industry is measuring the speedometer while ignoring the crash rate.

Every company in the table above can tell you how many PRs their AI tools generated. Almost none can tell you what happened to those PRs after they merged. Did they cause incidents? Did they get reverted? Did they require follow-up fixes? Did they achieve the intent they were built for?

The metrics that answer those questions exist. DORA added deployment rework rate as the 5th metric in 2024. LinearB tracks code-level rework. Graphite tracks review cycles. Change failure rate is a standard DORA metric every platform supports.

The tools are there. The measurement isn't happening. And until it does, the green arrows on the dashboard will keep hiding the delivery gap underneath.

## What To Measure Instead

Two numbers tell you whether your AI investment is working:

1. **Rework rate** — how often do changes need to be redone? Measured pre-merge (review rounds, patches) and post-merge (reverts, fix PRs, reopened tickets). This tells you whether the thinking happened before the building started.

2. **Change failure rate** — what percentage of changes cause production failures? This tells you whether your gates caught what the thinking missed.

If rework is climbing and change failure rate is climbing, your AI tools are making you faster at building the wrong thing. If both are flat or declining while throughput rises, your verification infrastructure is scaling with your generation speed.

That's the delivery gap, measured in two numbers that any team can track this week.

## How To Start Measuring This Week

The tools already exist. The problem was never tooling — it was that nobody connected the signals.

| Signal | How to measure it | Effort |
|--------|------------------|--------|
| **Post-merge rework (reverts)** | `git log --grep="Revert"` | 5 minutes |
| **Post-merge rework (DORA 5th metric)** | Faros, Swarmia (auto), or manual count of unplanned fix deploys | Already built if you use these platforms |
| **Pre-merge rework (review rounds)** | Graphite (built-in), Jellyfish (built-in), or one GitHub API script counting CHANGES_REQUESTED events | Built-in or one script |
| **Code churn** | LinearB (21-day window), GitClear (14-day), Pluralsight Flow (30-day) | Already built if you use these platforms |
| **Change failure rate** | Every DORA platform: Sleuth, Swarmia, Faros, LinearB | Already built |
| **Ticket reopens** | Jira "Reopening Counter" plugin, or Linear API query | One plugin install |
| **Abandonment rate** | `gh pr list --state closed` filtered by >7 days open, >50 lines changed, not merged | One script |
| **PR size** | Every platform, or `gh pr list --json additions,deletions` | Already built |

**Week 1:** Track reverts from git history and change failure rate from your existing DORA tooling. Two numbers. Takes an hour.

**Week 2:** Add pre-merge rework from your code review platform (Graphite, Jellyfish, or a GitHub API query). Now you have three numbers.

**Week 3:** Add ticket reopens from your issue tracker. Now you have four numbers covering pre-merge rework, post-merge rework, production escapes, and requirements rework.

**Week 4:** Run your first weekly review with all four numbers. Fifteen minutes, three people, one outlier, one control tweak.

You don't need a new platform. You need to read the numbers you already have — and ask what they say about whether AI is helping or hurting.
